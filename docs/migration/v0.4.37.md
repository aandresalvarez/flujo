This release introduces a project-centric workflow and improves the onboarding experience.

## Highlights

- `flujo init` scaffolds a new project with:
  - `flujo.toml` (includes `state_uri = "sqlite:///.flujo/state.db"`)
  - `pipeline.yaml` (minimal, valid pipeline)
  - `skills/` package (with `custom_tools.py`)
  - `.flujo/state.db` initialized on first use
- `flujo create` becomes a conversational wizard:
  - Prompts for goal (if omitted), pipeline name, and per-run budget
  - Writes `pipeline.yaml` and updates `[budgets.pipeline."<name>"]` in `flujo.toml`
  - Blocks side-effect skills unless `--allow-side-effects` or confirmation
- Project-aware defaults:
  - `flujo run` and `flujo validate` use the project `pipeline.yaml` when no path is provided
  - `flujo lens replay` uses the project `pipeline.yaml` by default if `--file` is omitted

## Migration Guidance

- Existing workflows continue to work; explicit file paths remain supported.
- Recommended path going forward:
  1. `mkdir my-bot && cd my-bot`
  2. `flujo init`
  3. `flujo create` (answer prompts)
  4. `flujo run --input "..."`
- If you previously used a global operations DB, either:
  - Keep using `FLUJO_STATE_URI` env var, or
  - Set `state_uri` in your project `flujo.toml`.

## Notes

- Inside a project, `flujo create` overwrites `pipeline.yaml` by default (no `--force` needed). For other `--output-dir` targets, `--force` is required to overwrite.
- The docs have been updated; user-guide paths are unified under `docs/user_guide/`.
